## Project Overview

CKYC (Central KYC) compliance platform. Operators generate search/upload files, submit them to the CKYC registry, and process responses. The backend orchestrates file generation via Python scripts, stores files in Cloudflare R2, and tracks batch lifecycle in PostgreSQL.

## Environment Setup

Copy `backend/.env.example` to `backend/.env`. All 22 variables are required ‚Äî the server exits at startup if any are missing (validated in `backend/src/config/env.js`).

Key services:
- **DATABASE_URL** ‚Äî Supabase PostgreSQL (`?sslmode=require` required)
- **REDIS_URL** ‚Äî Upstash Redis (`rediss://` with TLS)
- **R2_*** ‚Äî Cloudflare R2 storage (S3-compatible)
- **RESEND_API_KEY** ‚Äî transactional email
- **GMAIL_ADDRESS / GMAIL_APP_PASSWORD** ‚Äî IMAP access for Python scripts
- **PYTHON_SCRIPT_PATH** ‚Äî path to `backend/python/` folder
- **WORKERS_ENABLED** ‚Äî set `true` only when running BullMQ workers
- **FRONTEND_URL** ‚Äî used for CORS (`http://localhost:5173` locally)

Frontend uses `VITE_API_URL` (defaults to `/api/v1` via Vite proxy).

## Architecture

### Backend (`backend/src/`)

**Request flow:** `index.js` ‚Üí middleware (requestId ‚Üí JSON ‚Üí CORS ‚Üí logger) ‚Üí route ‚Üí controller ‚Üí service ‚Üí DB/R2/Python

- `config/env.js` ‚Äî single source of truth for all env vars; import `config` from here, never `process.env` directly
- `routes/` ‚Äî thin Express routers, no logic
- `controllers/` ‚Äî parse req, call service, send response
- `services/batch.service.js` ‚Äî core business logic for the entire batch lifecycle
- `services/r2.service.js` ‚Äî presigned URL generation for Cloudflare R2
- `services/audit.service.js` ‚Äî every significant action writes an audit log row
- `middleware/authenticate.js` ‚Äî JWT verification + token blacklist check (DB lookup per request)
- `middleware/requireRole.js` ‚Äî role guard used for admin routes
- `utils/ipcRunner.js` ‚Äî spawns Python scripts, sends JSON via stdin, reads JSON from stdout; payloads >64KB use a temp file at `/tmp/ckyc_ipc/`
- `constants/batchStatus.js` ‚Äî batch state machine and `assertValidTransition()` guard

**Workers** (`workers/`): BullMQ queues backed by Redis. Five queues: `search-generation`, `response-analysis`, `bulk-download`, `upload-generation`, `standalone-search`. Workers can run embedded (set `WORKERS_ENABLED=true`) or on-demand ‚Äî `launcher.js` spawns `runner.js` as a detached child process and tracks its PID in `/tmp/ckyc-workers/workers.pid`. `runner.js` auto-exits after idle (`WORKERS_IDLE_EXIT_MS`).

**Python IPC**: Node spawns `python3 <script>` with payload on stdin; Python writes a JSON result to stdout. Scripts live in `backend/python/`. PANs (personal tax IDs) are treated as PII ‚Äî `panSanitizer.js` redacts them from stderr before logging.

**Database** (Knex + raw `pg` pool):
- Migrations in `backend/migrations/` ‚Äî numbered sequentially
- The app queries via `pool` (raw SQL), not Knex query builder
- Knex is used only for migrations (`knex migrate:*`)

### Batch Lifecycle (state machine)

```
PROCESSING ‚Üí GENERATED ‚Üí DOWNLOADED ‚Üí WAITING_RESPONSE ‚Üí PROCESSING_RESPONSE ‚Üí COMPLETED
                                                                              ‚Üò FAILED ‚Üí PROCESSING (re-trigger)
```

- `PROCESSING`: search file being generated by Python
- `GENERATED`: search file in R2, ready for operator download
- `DOWNLOADED`: operator downloaded the search file for CKYC submission
- `WAITING_RESPONSE`: operator confirmed upload to CKYC registry
- `PROCESSING_RESPONSE`: response file uploaded, Python analysis running
- `COMPLETED`: download + upload output files ready in R2

### User Roles

- `operator` ‚Äî default; can manage batches
- `dev` ‚Äî admin; accesses `/admin/users` and `/admin/audit-logs`

Frontend enforces roles via `AdminRoute` in `App.tsx`. Backend enforces via `requireRole` middleware.

### Frontend (`frontend/src/`)

React + TypeScript + Vite. No backend-for-frontend ‚Äî calls the Express API directly.

- `stores/authStore.ts` ‚Äî Zustand store; persists JWT and user to `localStorage` (`ckyc_token`, `ckyc_user`)
- `api/client.ts` ‚Äî Axios instance with base URL from `VITE_API_URL`; attaches JWT via request interceptor; redirects to `/login` on 401
- `pages/` ‚Äî one file per route
- `components/ui/` ‚Äî shadcn/ui primitives (Button, Card, Dialog, etc.)
- `components/dashboard/` ‚Äî dashboard-specific composed components
- `lib/batchUtils.ts` ‚Äî batch display helpers
- `lib/safeStorage.ts` ‚Äî safe wrappers around `localStorage` (no-ops if unavailable)

### Deployment

- Backend ‚Üí Render (Express server + optional workers process)
- Frontend ‚Üí Cloudflare Pages (static Vite build)
- Storage ‚Üí Cloudflare R2
- DB ‚Üí Supabase (hosted PostgreSQL)
- Queue ‚Üí Upstash Redis

### API Routes

All routes are prefixed `/api/v1/`.

| Prefix | Auth | Purpose |
|--------|------|---------|
| `/auth` | public / bearer | login, logout |
| `/batches` | bearer | batch CRUD, file URLs, response processing |
| `/admin` | bearer + `dev` role | user management, audit logs |
| `/training` | bearer | training video list |
| `/internal` | cron secret header | automated daily batch trigger |
